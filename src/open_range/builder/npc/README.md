# NPC System

The NPC (Non-Player Character) system generates realistic green-team
activity inside the cyber range.  Each NPC has a persona, personality,
daily task list, and episodic memory.  NPCs observe events, make
personality-driven decisions about when to act, and communicate via
email or chat.

## Module overview

| Module | Purpose |
|--------|---------|
| `identity.py` | Persona models (`NPCBackstory`, `NPCPersonality`, `NPCProfile`) and profile generation |
| `persona.py` | Default persona definitions (8 built-in employees) |
| `memory.py` | Episodic memory stream with recency/importance/relevance scoring |
| `tasks.py` | Daily task templates and `generate_tasks()` |
| `email_templates.py` | Deterministic email/reply content generation |
| `outbox.py` | Bridge components: `ActionOutbox`, `EventInbox`, `SimClock`, `MessageStore` |
| `runtime_agent.py` | Core agent loop, observation system, action building |
| `npc_manager.py` | Orchestrates agent lifecycle (start/stop) |
| `planner.py` | Daily planner used by the offline scheduler |
| `actions.py` | Action definitions shared with the green scheduler |

The green scheduler (`open_range/green.py`) coordinates NPC actions
with the runtime in both offline (scripted) and online (agent-driven)
modes.

## Architecture

### Persona model

Every NPC is a `GreenPersona` enriched with an `NPCProfile`:

```
GreenPersona
  id, role, department, awareness, susceptibility
  profile: NPCProfile
             backstory: NPCBackstory
               full_name, location, working_hours
               communication_style, preferred_modality (email|chat)
               friends, projects, responsibilities, background
             personality: NPCPersonality
               mood        (focused|anxious|relaxed|distracted|bored)
               disposition (cooperative|competitive|cautious|impulsive)
               work_ethic  (diligent|average|lazy)
               interpersonal_style (formal|casual|terse|verbose)
               risk_tolerance  (0.0-1.0)
               chattiness      (0.0-1.0)
```

Profiles are generated deterministically from role-keyed template pools
by `identity.generate_profiles()`.

### Memory stream

`MemoryStream` stores entity-relationship triples:

```
MemoryEntry(subject, relation, object_, importance, timestamp, tags)
```

Retrieval uses a composite score:

```
score = w_recency * recency(t) + w_importance * (importance/10) + w_relevance * jaccard(query, tags)
```

- **Recency**: exponential decay (0.9992^elapsed_seconds)
- **Importance**: 1-10 scale normalized to 0-1
- **Relevance**: Jaccard overlap of query tags against memory tags + tokenized triple fields

Max capacity is 150 entries; overflow evicts the lowest-importance entry.

### Daily tasks

`generate_tasks()` produces a time-ordered list of `NPCTask` objects
from role-keyed template pools (IT, exec, sales, generic).  Email tasks
get content generated by `email_templates.generate_email_content()`,
grounded in the sender's current task context and the recipient's role.

Tasks carry a `modality` field ("email" or "chat") derived from the
persona's `preferred_modality`.

## Observation system

The observation system decouples event arrival from action.  Instead of
reacting to every event immediately, NPCs record observations and
decide when to act based on personality.

### Observation categories

When an event arrives, `_create_observation()` classifies it:

| Category | Condition | Importance | Path |
|----------|-----------|------------|------|
| `security` | Malicious + high-severity (InitialAccess, CredentialObtained, UnauthorizedCredentialUse) | 8.0 | Fast path (immediate reaction) |
| `alert` | Other malicious events, or DetectionAlertRaised | 5.0 | Deferred |
| `message` | BenignUserAction targeting this NPC, first time from sender | 3.5 | Deferred |
| *(none)* | Own events, routine service activity, already-read senders | -- | Ignored |

### Personality-driven check propensity

`_check_propensity()` maps personality traits to a 0.05-0.95
probability of checking pending observations on any given tick:

```
propensity = 0.3 (base)
           + mood_weight        (distracted: +0.30 ... focused: -0.10)
           + work_ethic_weight  (lazy: +0.20 ... diligent: -0.15)
           + style_weight       (verbose: +0.10 ... terse: -0.05)
           + 0.20 * chattiness
```

Example values for built-in personas:

| Persona | Mood | Work ethic | Style | Chattiness | Propensity |
|---------|------|------------|-------|------------|------------|
| Bob Smith | focused | diligent | terse | 0.3 | ~0.06 |
| Janet Liu | relaxed | average | casual | 0.9 | ~0.53 |
| Dan Wu | focused | diligent | verbose | 0.8 | ~0.21 |
| Priya Patel | relaxed | diligent | casual | 0.7 | ~0.29 |

### Observation aging

Stale observations become harder to ignore.  An age boost of
`min(0.3, age_minutes / 60)` is added to propensity, so even a
focused NPC with propensity 0.06 will check after ~18 minutes
(0.06 + 0.3 = 0.36 effective).

### Between-task batch checking

When the agent completes a scheduled task, `_between_tasks` is set to
`True`.  This forces `_should_check_observations()` to return `True`
on the next tick, simulating "I finished my task, let me check
messages."  This is the primary mechanism by which focused/diligent
NPCs process their inbox.

## Action priority

`_maybe_submit_next()` uses a strict priority order:

```
1. Security reactions   (_reaction_queue, fast path)
2. Pending observations (personality-gated, produces reactions)
3. Scheduled tasks      (from daily task list)
```

## Communication modality

NPCs communicate via email or chat.  The modality affects:

- **Service routing**: email -> `svc-email`, chat -> `svc-chat`
- **Subject lines**: chat messages carry no subject
- **Reply medium**: replies match the incoming message's modality
- **Social messages**: always use chat (casual by nature)

## LLM integration

When a model is configured (via constructor or `OPENRANGE_NPC_MODEL`
env var), the agent uses LLM calls for:

- Email composition (`_build_action_llm`)
- Reply composition (`_llm_compose_reply`)
- Security reaction decisions (`_llm_decide_reaction`)

All LLM calls fall back to deterministic templates on failure.

## Running the visualization

```bash
# Offline mode (deterministic, fast)
python scripts/npc_activity.py --minutes 480 --timestamps

# Online mode (agent-driven, observation system active)
python scripts/npc_activity.py --online --minutes 480 --timestamps

# Online with LLM-generated email content
python scripts/npc_activity.py --online --minutes 480 --timestamps --model gpt-4o-mini
```
