# OpenRange Training Data Spec

This document defines the branch-native training-data contract for OpenRange
V1.

## Purpose

OpenRange training data must come from:

- admitted snapshots
- executed traces over those snapshots
- full world/runtime metadata

It must not primarily come from:

- generic cyber chat corpora
- hand-written transcripts
- old-format traces that do not match the current runtime

## Source Types

OpenRange distinguishes two trace sources:

- `sim`: cheap bootstrap traces generated by reference replay through the public decision loop
- `runtime`: realism-critical traces generated from actual runtime episodes over admitted snapshots

These sources must remain explicitly labeled and must not be silently mixed.

## Row Contract

The canonical export unit is one decision row per external actor decision.

Each row must carry:

- `trace_source`
- `action_source`
- `split`
- `snapshot_id`
- `world_id`
- `world_hash`
- `lineage`
- `episode_id`
- `mode`
- `start_state`
- `role`
- `decision_index`
- `observation`
- `chosen_action`
- `chosen_action_text`
- `emitted_events`
- `result_stdout`
- `result_stderr`
- `grounded_effects`
- `mitigation_effects`
- `reward_delta`
- `winner`
- `terminal_reason`
- `build_config`
- `episode_config`
- `weaknesses`
- `benchmark_tags`

This is the ground-truth training substrate.

Default prompt/rendering rule:

- exported row metadata may include weakness inventory and benchmark tags
- default SFT/eval prompts must mirror the runtime observation surface, with completions rendered as concrete `Action` JSON
- hidden weakness inventory, benchmark tags, snapshot ids, lineage ids, split labels, action-source labels, and other evaluator-only context must be excluded from the default prompt text unless an explicit research/debug export asks for them

## Action Contract

Training should happen on the actual OpenRange runtime action contract:

- observe the current runtime state
- emit the next concrete `Action`
- score the resulting grounded effects and mitigation effects

OpenRange V1 should not treat unconstrained free-form generation as the primary learning objective.

## Action Sources

Each exported decision row includes:

- one chosen runtime action
- the source of that action, such as `reference_runtime` or `reference_sim`

This keeps trace export grounded in the public runtime loop while distinguishing live runtime rows from sim-derived bootstrap rows.

## Role Coverage

The trace generator must cover:

- `red_only`
- `blue_only_live`
- `blue_only_from_prefix`
- optional `joint_pool`
- optional `sim` bootstrap traces

Blue traces are first-class. They are not deferred until after red.

## Split Policy

Splits must be assigned by lineage root, not random row.

Required properties:

- parent/child worlds stay in the same split
- rows from the same lineage root do not cross train/eval boundaries
- split metadata is stored on each row

## Action Semantics

Trace export must keep blue mitigation semantics honest.

If a control action is not genuinely service-native patching, the exported action should prefer:

- `contain`
- `mitigate`
- `break_path`

over claiming a full `patch`.

## Derived Views

The canonical trace rows may be transformed into training views, including:

- `decision_sft.jsonl` for small causal-LM warmup over observation-to-action completions
- ranking/preference datasets

Derived views must preserve snapshot and lineage metadata.

The default export should also write clean shards for:

- `red` vs `blue`
- `runtime` vs `sim`
- role-only all-source views such as `sft_red_all.jsonl`
- role-and-source views such as `sft_blue_runtime.jsonl`

## Default Training Sequence

The default branch-native learning sequence is:

1. generate raw trace rows from admitted snapshots
2. generate branch-native action-completion SFT rows
3. train small models on that branch-native data
4. evaluate in-loop on admitted snapshots and mutations
5. use larger-scale RL only after those stages

Default tiny-train behavior:

- default on-the-fly dataset generation is `runtime`-only
- generated corpora should be regenerated from the current code/runtime contract,
  not silently reused from stale cache directories
